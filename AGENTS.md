# AGENTS.md

## 项目目标

- 本项目是本地优先的 Codex Skill 扫描、审计、路由和测试工具。
- 第一版只做 `scan`、`audit`、`route` 和 `eval`。
- 不做 SaaS、登录、数据库和网页后台。
- 本项目只提供本地路由预测，不声称能够控制或完全知道 Codex 的内部决策。

## 用户背景

- 项目负责人不懂代码。
- 项目负责人不负责手写代码。
- 项目负责人负责提供产品想法、判断功能是否有价值、进行实际体验和验收。
- Codex 负责技术方案、编写代码、运行测试、修复问题和维护文档。
- 所有技术说明必须使用通俗中文。
- 不能只给出复杂源码作为解释。
- 不要默认项目负责人能理解源代码、命令行报错或复杂技术术语。
- 每次任务结束必须提供人工验收步骤。

## 开发方式

- 每次只实现当前明确指定的一个阶段。
- 不提前开发后续功能。
- 开始工作前先读取产品需求和相关文件。
- 修改前检查当前 Git 状态。
- 只修改与当前任务相关的文件。
- 新增依赖前必须说明理由。
- 优先使用简单、成熟、稳定的方案。
- 不为了展示技术而增加复杂架构。
- 当前代码阶段只允许实现明确指定的小步骤，优先从最小可运行 CLI 入口开始。

## 测试规则

- 所有核心功能必须有测试。
- 修改后必须实际运行测试。
- 后续开始写代码后，必须运行类型检查和构建。
- 不允许通过删除测试解决失败。
- 不允许偷偷降低验收标准。
- 不允许只声称测试通过而不实际执行。
- 发现失败必须如实汇报。
- 当前代码阶段需要优先使用 Node.js 内置测试能力验证 CLI 入口，不新增不必要依赖。

## 文件安全

- 不修改与当前任务无关的文件。
- 不覆盖用户已有内容。
- 不删除用户文件。
- 不上传私人数据。
- 不提交密钥和环境变量。
- 自动生成内容应使用明确标记。
- 危险操作必须提前说明。
- 涉及写入用户配置时必须默认 dry-run。
- 第一版的 `scan`、`audit`、`route` 和 `eval` 默认只读，不得修改用户 Skill。

## 沟通规则

每次完成任务后，必须使用通俗中文说明：

1. 完成了什么；
2. 修改了哪些文件；
3. 为什么这样处理；
4. 是否运行了测试；
5. 测试结果；
6. 用户如何验收；
7. 已知限制；
8. 风险；
9. 推荐的下一步。

不要自动执行下一步，除非用户明确要求。

## Git 规则

- 每个阶段尽量保持修改范围清晰。
- 不创建包含多个无关功能的大提交。
- 提交前检查 Git diff。
- 不提交私人路径、密钥、缓存或构建产物。
- 未经用户要求，不执行强制推送。
- 未经用户要求，不重写 Git 历史。
- 未经用户要求，不执行 Git 提交。

## 产品原则

- 本地优先：默认不上传 Skill 内容、项目路径、用户任务、测试数据和项目代码。
- 默认只读：第一版不得修改用户 Skill。
- 可解释：不能只输出评分，必须说明推荐和排除原因。
- 不伪装确定性：必须区分本地路由预测和 Codex 实际内部调用结果。
- 不依赖外部 AI：第一版必须在没有 API Key、云服务或外部大模型的情况下运行。
- 不追求功能数量：第一版优先保证四个命令稳定可靠。
