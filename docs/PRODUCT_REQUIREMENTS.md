# Codex Skill Router 产品需求文档

## 1. 项目概述

Codex Skill Router 是一个本地优先的开源工具，暂定命令行简称为 `csr`。

它帮助 Codex 用户检查、测试和改进 Skills 的自动选择效果。第一版只围绕四件事展开：

- 扫描当前有哪些 Skills；
- 检查 Skill 配置是否清楚、完整、容易被正确触发；
- 根据用户输入的任务，预测哪些 Skills 更适合；
- 用测试任务检查路由结果是否符合预期。

本项目不是重新开发 Codex，也不试图控制 Codex 内部决策。它提供的是本地预测、质量检查和可解释建议。

## 2. 背景和问题

用户安装多个 Codex Skills 后，常见问题包括：

- 不知道当前安装了哪些 Skills；
- 不知道不同 Skills 分别适合什么任务；
- 多个 Skills 的功能可能重叠；
- Skill 的 description 写得太宽泛，可能导致误调用；
- Codex 可能漏掉本应使用的 Skill；
- 修改 Skill 后，不知道路由效果是否变好；
- 缺少一种简单方法来测试“某个任务应该选择哪些 Skills”。

因此，本项目希望形成一条简单流程：

```text
扫描 Skills
-> 检查配置质量
-> 预测任务应该使用哪些 Skills
-> 测试路由是否正确
-> 找出误报、漏报和功能重叠
-> 给出可解释的改进建议
```

## 3. 产品定位

一句话定位：

```text
帮助用户检查、测试和改进 Codex Skills 的自动选择效果。
```

更准确的产品方向：

```text
Skill Doctor
+ Explainable Router
+ Routing Eval
```

产品目标：

- 提高 Skill 配置质量；
- 预测哪些 Skills 更适合当前任务；
- 解释为什么推荐或不推荐某个 Skill；
- 测试 Skill 路由是否符合预期；
- 修改配置后进行回归测试。

## 4. 目标用户

### 用户一：安装多个 Skills 的 Codex 用户

这类用户希望知道自己装了什么、每个 Skill 适合什么任务，以及为什么某些任务没有触发预期 Skill。

### 用户二：Skill 作者

这类用户希望在发布或修改 Skill 前，检查 description 是否清楚，是否会和其他 Skill 冲突，是否会引入路由回归。

### 用户三：维护 Skill 仓库的团队

这类用户需要统一检查规则、批量发现冲突，并在未来接入 CI 做自动测试。

第一版优先服务：

```text
个人 Codex 用户
+
开源 Skill 作者
```

企业团队只作为后续方向。

## 5. 用户痛点

个人 Codex 用户的痛点：

- 不知道有哪些 Skills；
- 不知道哪个 Skill 应该处理什么任务；
- 不知道是否发生功能重叠；
- 不知道 Codex 为什么选错或没选。

Skill 作者的痛点：

- 不知道 description 是否写得清楚；
- 不知道新增 Skill 是否会抢占其他 Skill；
- 不知道修改后是否引入路由回归；
- 需要发布前质量检查。

团队维护者的痛点：

- Skill 数量较多；
- 需要建立统一检查规则；
- 需要在 CI 中测试路由；
- 需要清楚误报、漏报和冲突。

## 6. 核心使用场景

### 场景一：扫描当前 Skills

用户希望知道当前项目目录和用户目录中安装了什么 Skills。工具应列出名称、描述、路径和来源。

### 场景二：发现 description 问题

用户希望知道为什么某个 Skill 经常无法自动触发。工具应指出 description 是否过短、过宽、缺少使用时机，或缺少“不适用场景”。

### 场景三：发现 Skill 重叠

用户安装了多个 UI 或浏览器相关 Skill，希望知道是否重复。工具应提示可能重叠的 Skill，并用通俗语言说明原因。

### 场景四：输入任务并获得推荐

用户输入：

```text
优化亚马逊选品工具的页面结构，并检查移动端。
```

工具应推荐可能相关的业务 Skill、UI Skill 和浏览器验证 Skill，并解释推荐依据。

### 场景五：测试路由回归

Skill 作者修改 description 后，运行 Eval 判断准确率是提高还是下降。

### 场景六：错误和异常

工具应能处理找不到 Skills、SKILL.md 不存在、YAML 格式错误、缺少 name、缺少 description、Skill 名称重复、Eval 文件错误、用户输入为空、路径没有权限等情况，并给出非程序员也能理解的提示。

## 7. 第一版四个核心命令

第一版只考虑以下四个命令：

- `csr scan`
- `csr audit`
- `csr route`
- `csr eval`

第一版不增加网页界面、登录、数据库、云服务或自动修复功能。

## 8. 每个命令的输入

### `csr scan`

输入：

- 可选的扫描路径；
- 默认扫描项目级 Skills 和用户级 Skills。

示例：

```bash
csr scan
```

### `csr audit`

输入：

- 可选的扫描路径；
- 可选的输出格式；
- 默认检查当前可发现的 Skills。

示例：

```bash
csr audit
```

### `csr route`

输入：

- 用户的一段真实任务描述；
- 可选的扫描路径；
- 可选的输出格式。

示例：

```bash
csr route "优化现有的 Next.js 页面，并检查移动端显示"
```

### `csr eval`

输入：

- 一组预先准备的测试任务；
- 每个测试任务包含 prompt、必须包含的 Skills、必须排除的 Skills；
- 可选的输出格式。

示例：

```bash
csr eval
```

测试任务示例：

```yaml
prompt: "优化现有后台页面，并检查移动端"
expected:
  include:
    - frontend-ui-review
    - browser-validation
  exclude:
    - database-migration
```

## 9. 每个命令的输出

### `csr scan`

输出：

- Skill 名称；
- description；
- 路径；
- 来源是项目级还是用户级；
- 是否存在基础格式错误。

### `csr audit`

输出：

- 缺少 name 的 Skills；
- 缺少 description 的 Skills；
- description 过短、过长或过于宽泛的 Skills；
- 没有说明使用时机的 Skills；
- 没有说明不适用场景的 Skills；
- 名称重复的 Skills；
- 可能高度重叠的 Skills；
- 通俗的修复建议。

### `csr route`

输出：

- 推荐的 Skills；
- 不推荐的相似 Skills；
- 为什么推荐；
- 为什么没有推荐某些相似 Skill；
- 哪些条件影响判断；
- 明确提示：这是本地路由预测，不代表 Codex 内部一定会实际调用。

示例：

```text
推荐：
- frontend-ui-review
- browser-validation

不推荐：
- backend-security
- database-migration
```

### `csr eval`

输出：

- 总测试数；
- 完全正确数；
- 部分正确数；
- 错误数；
- 必须调用 Skill 的命中率；
- 不应调用 Skill 的正确排除率；
- 误报；
- 漏报；
- 平均推荐 Skill 数量；
- 失败案例和原因。

## 10. 每个命令的异常情况

所有异常提示都应使用通俗中文，避免只输出技术报错。

### `csr scan`

异常情况：

- 找不到 Skills；
- 路径不存在；
- 路径没有权限；
- SKILL.md 不存在；
- 文件无法读取。

建议提示方向：

```text
没有找到可扫描的 Skills。请确认当前目录或用户 Skills 目录是否存在。
```

### `csr audit`

异常情况：

- SKILL.md 格式错误；
- 缺少 name；
- 缺少 description；
- description 无法解析；
- Skill 名称重复。

建议提示方向：

```text
这个 Skill 缺少 description，工具无法判断它适合什么任务。
```

### `csr route`

异常情况：

- 用户输入为空；
- 没有可用 Skills；
- 所有 Skills 都缺少有效 description；
- 推荐结果为空；
- 路径没有权限。

建议提示方向：

```text
这次没有找到足够可靠的推荐结果。可能是任务描述太短，或 Skill 描述不够清楚。
```

### `csr eval`

异常情况：

- Eval 文件不存在；
- YAML 格式错误；
- 测试任务缺少 prompt；
- expected.include 或 expected.exclude 格式错误；
- 某些 expected Skill 不存在；
- 没有任何测试任务。

建议提示方向：

```text
测试文件中有一条任务缺少 prompt，请补充用户任务描述后再运行。
```

## 11. 第一版范围

第一版必须优先做到：

- 本地运行；
- 功能清晰；
- 安全只读；
- 结果可解释；
- 容易测试；
- 容易安装；
- 别人能够真实使用。

第一版只交付 `scan`、`audit`、`route`、`eval` 四个核心能力。

## 12. 明确不做的功能

以下功能不属于第一版范围：

- 用户登录；
- 云端服务器；
- 数据库；
- 付费系统；
- SaaS 后台；
- 网页管理界面；
- Skill 市场；
- 自动下载大量 Skills；
- 团队协作；
- 多用户权限；
- 多模型支持；
- Claude Code 适配；
- Gemini CLI 适配；
- GitHub Copilot 适配；
- MCP Server；
- 自动控制 Codex 内部调用；
- 复杂向量数据库；
- 依赖外部大模型才能运行；
- 自动修改所有用户 Skill；
- 复杂安全扫描引擎。

## 13. 产品原则

### 本地优先

默认不上传以下内容：

- Skill 内容；
- 项目路径；
- 用户任务；
- 测试数据；
- 项目代码。

### 默认只读

第一版的 `scan`、`audit`、`route` 和 `eval` 不得修改用户 Skill。

未来如果加入修改功能，必须：

- 默认 dry-run；
- 先展示 diff；
- 用户明确确认后再写入；
- 修改前创建备份。

### 可解释

不能只输出一个评分。必须解释：

- 哪些信息触发了推荐；
- 哪些排除条件降低了评分；
- 为什么同类 Skill 没有被选中；
- 当前判断有哪些局限。

### 不伪装确定性

工具必须明确区分：

```text
本地路由预测
```

和：

```text
Codex 实际内部调用结果
```

不能声称可以百分之百控制或知道 Codex 的内部决策。

### 不依赖外部 AI

第一版必须在没有 API Key、没有云服务、没有外部大模型的情况下运行。

后续可以加入可选 AI 增强，但必须保留非 AI 的基础模式。

### 不追求功能数量

第一版优先保证四个命令稳定可靠，不提前增加大型功能。

## 14. 用户安全和隐私

安全和隐私要求：

- 默认不上传用户 Skill；
- 默认不上传用户项目路径；
- 默认不上传用户输入的任务；
- 默认不上传测试数据；
- 默认不修改用户 Skill；
- 涉及写入时必须默认 dry-run；
- 输出中应避免泄露私人路径、密钥、环境变量和缓存内容；
- 报告中如果需要展示路径，应允许用户选择简化路径显示。

## 15. 成功标准

### 产品成功

- 用户能够知道自己安装了哪些 Skills；
- 用户能够理解 Skill 配置中的主要问题；
- 用户能够输入任务并得到可解释的推荐；
- 用户能够使用测试集检查路由变化；
- 工具不会默认修改用户 Skill；
- 输出对非程序员也较容易理解。

### 工程成功

本次只规划，不实施。未来工程目标记录如下：

- 使用 TypeScript；
- Node.js 20+；
- 可以通过 npm 或 npx 使用；
- 支持 Windows、macOS 和 Linux；
- 核心模块有测试；
- GitHub Actions 能自动运行测试；
- 支持终端输出和 JSON 输出；
- 涉及写入时默认 dry-run；
- README 中的命令必须真实可执行。

## 16. 第一版验收标准

未来满足以下条件后，才可以发布 `v0.1.0`：

- `scan` 可以真实扫描 Skills；
- `audit` 可以发现基础问题；
- `route` 可以推荐并解释；
- `eval` 可以运行测试集；
- 核心测试通过；
- 类型检查通过；
- 构建通过；
- 没有密钥或私人路径；
- README 完整；
- 至少有一套演示样本；
- 至少有 30 条路由测试任务。

## 17. 已知风险

1. 本地 Router 推荐不等于 Codex 实际调用。
   缓解方向：在输出中明确标注“本地预测”，不承诺控制 Codex。

2. 仅靠规则和关键词可能出现误判。
   缓解方向：保留解释、误报和漏报报告，后续再考虑可选 AI 增强。

3. 中文和英文 Skill 描述可能增加判断难度。
   缓解方向：第一版记录语言特征，后续优化多语言规则。

4. Skill 标准和 Codex 机制可能变化。
   缓解方向：把规则写成可维护配置，并在文档中说明版本限制。

5. Skill 数量很多时可能出现性能和上下文问题。
   缓解方向：第一版优先做清晰扫描，后续再优化缓存和分页输出。

6. 自动修复 description 可能改变原 Skill 真实意图。
   缓解方向：第一版不做自动修复；未来必须 dry-run、展示 diff 并备份。

7. 第一版不调用外部 AI，因此语义判断能力有限。
   缓解方向：明确基础模式能力边界，后续增加可选增强但不强依赖。

8. 不同用户的 Skill 目录和系统环境可能不同。
   缓解方向：支持清楚的路径提示和可配置扫描路径。

9. Windows、macOS 和 Linux 路径行为可能不同。
   缓解方向：未来测试覆盖三类系统，并避免硬编码路径。

10. 项目可能被误解为可以强制控制 Codex。
    缓解方向：在 README、命令输出和文档中反复说明它只是本地预测工具。

## 18. 技术难点

本节只记录未来开发会遇到的难点，不在本次实现。

- 如何准确发现不同位置的 Skills；
- 如何兼容不同格式的 SKILL.md；
- 如何判断 description 是否过宽或过窄；
- 如何识别多个 Skill 的功能重叠；
- 如何在不依赖外部 AI 的情况下做较好的任务匹配；
- 如何让终端输出既适合普通用户阅读，又适合自动化工具解析；
- 如何设计稳定的 Eval 评分口径。

## 20. 待确认问题

以下问题会影响后续开发，建议在正式写代码前确认：

1. 第一版默认扫描哪些 Skill 目录。
   建议方案：先支持当前项目目录和用户 `.codex/skills` 目录，再允许用户手动指定路径。

2. 第一版终端输出使用中文还是英文。
   建议方案：优先中文输出，同时保留 JSON 输出，方便后续英文文档和自动化使用。

3. 第一版是否需要直接读取插件缓存中的 Skills。
   建议方案：先只读本地可访问路径，不做自动下载，也不访问网络。

4. Eval 测试文件的默认位置。
   建议方案：未来使用 `examples/eval.yml` 或项目根目录下的明确配置文件，但本次不创建样例文件。
